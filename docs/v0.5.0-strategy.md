# v0.5.0 Strategy

## Purpose

Decide the best `v0.5.0` direction without adding runtime features yet.

## Current System State

The repository now has four stable layers:

- foundation: markdown-first operating contract, command loop, skills, workflows, patterns, templates
- design system: harness-build, profile templates, team architecture selection, memory model, SDLC model, skill authoring system, demo harness build
- validation: target profile validation, target goal validation, dependency-free contract checks
- ergonomics: install guidance, install output examples, tiny-repo example, small-repo memory conventions, walkthroughs, validation troubleshooting
- distribution question: the repo now needs clearer source-pack versus target-repo positioning

## Strategic Options

### Option A: Plugin Distribution Model

Focus:

- clarify the harness as a plugin-like capability pack
- define source repo versus target repo semantics
- document current and future consumption modes

Pros:

- reduces product-positioning confusion
- strengthens adoption clarity
- aligns the repo with superpowers-style usage

Cons:

- docs-heavy and conceptual
- does not add immediate workflow automation

### Option B: More Dogfood On Real Repos

Focus:

- adopt the harness into `2-3` small real repos
- capture friction
- improve examples based on evidence

Pros:

- highest confidence
- avoids speculative features
- keeps the project grounded

Cons:

- slower
- requires manual effort

### Option C: Automation-Lite

Focus:

- small helper command or script to create starter `.harness/` profile files from templates
- still no runtime or server layer

Pros:

- improves adoption speed
- reduces manual setup friction

Cons:

- can drift toward generator complexity
- must avoid becoming a framework

### Option D: Memory Conventions v2

Focus:

- richer markdown memory conventions
- project facts, decisions, root causes, reusable commands, hazards

Pros:

- strengthens the dev-memory inspiration
- helps long-running projects

Cons:

- may be too abstract without more distribution clarity

### Option E: Runtime-Specific Usage Polish

Focus:

- stronger guides for Claude Code, Cursor, Codex, Gemini CLI, and OpenCode

Pros:

- useful for real users
- low implementation risk

Cons:

- runtime docs can become stale
- less core architecture value

### Option F: Public / Community Polish

Focus:

- contribution examples
- issue templates
- example PR workflow
- public docs cleanup

Pros:

- improves open-source usability

Cons:

- less core harness value

## Recommendation

Recommend Option A first: Plugin Distribution Model.

Reason:

The project already has enough structure and enough ergonomics work to expose a positioning problem: users can still read the repository as “clone this and work in it.” Clarifying the harness as a capability pack should come before more automation or more dogfood layers.

## v0.5.0 Proposed Target

`v0.5.0` = Plugin Distribution Model

Goal:

Clarify how the harness should be consumed as a plugin-like markdown capability pack, define source-repo versus target-repo boundaries, and document future distribution channels without implementing them yet.

## Scope

- plugin-like positioning
- distribution model docs
- consumption modes
- installed surface clarity
- future runtime and plugin-registry boundaries

## Non-Goals

- no generator yet
- no runtime adapters
- no server or database systems
- no semantic validation
- no deep scanning
- no marketplace automation
- no package publishing automation

## Success Criteria

- plugin-like consumption is explicit in the target and README
- source repo versus target repo distinctions are clear
- distribution model docs exist
- future plugin or registry ideas stay documented as boundaries, not implementations
- no runtime or automation scope creep is introduced

Add or update [docs/v0.5.0-plan.md](v0.5.0-plan.md) focused on the Plugin Distribution Model.
