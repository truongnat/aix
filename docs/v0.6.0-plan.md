# v0.6.0 Plan

## Purpose

Plan `v0.6.0` as a runtime-consumption-guides release that explains how to use the capability pack across common AI coding runtimes.

## Target Summary

`v0.6.0` should explain how agents in common runtimes consume the plugin-like markdown pack while preserving the markdown-first, runtime-light model.

## Scope

- clarify runtime consumption model
- update runtime docs for plugin-like pack usage
- add runtime-specific quick start prompts
- add runtime-specific what-the-agent-should-read guidance
- add runtime-specific safety boundaries
- add runtime-specific validation flow
- keep all guidance docs-only

## Non-Goals

- no runtime adapters
- no extension or plugin implementation
- no marketplace package
- no package publishing automation
- no server, database, or background systems
- no semantic validation

## Runtime Targets

- Claude Code
- Cursor
- Codex
- Gemini CLI
- OpenCode

## Recommended Implementation Order

1. runtime consumption overview
2. Claude Code consumption guide
3. Cursor consumption guide
4. Codex consumption guide
5. Gemini CLI consumption guide
6. OpenCode consumption guide
7. compare runtime patterns
8. readiness and release

Current step:

- Step 1 adds the runtime consumption overview
- Step 2 updates the Claude Code guide for capability-pack consumption
- Step 3 updates the Cursor guide for capability-pack consumption
- Step 4 updates the Codex guide for capability-pack consumption
- Step 5 updates the Gemini CLI guide for capability-pack consumption
- Step 6 updates the OpenCode guide for capability-pack consumption
- Step 7 adds the runtime comparison guide

## Definition Of Done

- runtime consumption model exists
- runtime docs are reframed around capability-pack consumption
- runtime-specific quick start prompts exist
- runtime-specific validation flow is documented
- `node validate.js` passes
- `npm test` passes

## Release Risks

- runtime docs becoming repetitive instead of clarifying differences
- drifting into adapter design instead of usage guidance
- runtime docs going stale if they over-assume runtime-specific features

## Safety Boundaries

- keep markdown as the source of truth
- keep all guidance docs-only
- do not add runtime adapters or integrations
- do not add marketplace or package publishing automation
- do not add semantic validation or deep scanning
