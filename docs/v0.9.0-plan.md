# v0.9.0 Plan

## Purpose

Plan **Plugin Install UX** for `ai-engineering-harness` so the pack installs like a plugin or superpower—without requiring consumers to manually clone the source repository.

## Pivot Note

Git tag `v0.9.0` shipped **contract-freeze documentation** (frozen `docs/frozen-*.md`, index, breaking-change policy). That work remains useful as **draft / pre-v1 contract candidates**.

Consumer-facing target was wrong for v1: freezing a **local source-pack install** workflow does not match plugin-style adoption. Active roadmap work continues under this plan as **Plugin Install UX** (next ship likely `v0.9.1` or later patch; see [CHANGELOG](../CHANGELOG.md)).

## Target Summary

`v0.9.x` should let a user run **one command** to install the capability pack into the current repo or a chosen target—then init/validate—without `git clone` of `ai-engineering-harness` as a required step.

## Scope

- `install.sh` one-line remote installer (Step 2+)
- optional `bin/ai-harness.js` global CLI
- keep `install.js` as canonical copy implementation
- plugin install UX docs and security notes
- one-line installer design
- dogfood one-line install
- README and TARGET aligned with plugin-style consumption
- mark frozen contract docs as pre-v1 candidates until plugin UX is dogfooded

## Non-Goals

- npm package publishing
- marketplace automation
- runtime adapters or MCP
- semantic validation
- replacing `install.js` copy behavior
- breaking existing `node install.js --target` for maintainers who clone the source pack

## Desired Install UX

Target consumer flow:

```bash
curl -fsSL https://raw.githubusercontent.com/truongnat/ai-engineering-harness/main/install.sh | sh
curl -fsSL https://raw.githubusercontent.com/truongnat/ai-engineering-harness/main/install.sh | sh -s -- --target .
curl -fsSL https://raw.githubusercontent.com/truongnat/ai-engineering-harness/main/install.sh | sh -s -- --target ../my-project --dry-run
```

Optional global flow:

```bash
curl -fsSL https://raw.githubusercontent.com/truongnat/ai-engineering-harness/main/install.sh | sh -s -- --global
ai-harness install --target .
ai-harness validate --target .
```

Maintainer flow (still supported, not required for consumers):

```bash
git clone https://github.com/truongnat/ai-engineering-harness.git
cd ai-engineering-harness
node install.js --target ../my-project
```

## Install Modes

| Mode | Description |
|---|---|
| Project default | `--target` defaults to current working directory |
| Explicit target | `--target <path>` |
| Dry run | `--dry-run` forwards to `install.js` |
| Force overwrite | `--force` forwards to `install.js` |
| Global CLI | `--global` installs `ai-harness` shim (design Step 2+) |
| Pin version | `--ref` / `--version` downloads specific tag (design) |

## CLI Commands

Planned `ai-harness` commands (dependency-free Node shim):

- `ai-harness install --target <path>` [--dry-run] [--force]
- `ai-harness validate --target <path>` [--profile-only] [--goal <id>]
- `ai-harness init` (future: scaffold `.harness/` in target)

Validation continues to use source-pack `validate.js` until in-target copy is an explicit future contract.

## Definition Of Done

- consumer can install without manual clone (one-line `install.sh`)
- `install.js` behavior unchanged for maintainers
- plugin UX and security documented
- one-line install dogfooded in at least one external target repo
- `node validate.js` and `npm test` pass
- frozen contract docs labeled pre-v1 until plugin UX ships

## Release Risks

- `curl | sh` trust model requires clear security docs and tag pinning
- GitHub tarball download breaks offline or rate-limited environments
- global CLI path differences across OS shells
- freezing contracts before plugin UX would cement the wrong adoption story

## Safety Boundaries

- no secrets, telemetry, or hidden network beyond downloading the pack
- install script must not modify files outside target (and optional global bin dir)
- markdown remains source of truth
- dependency-free Node and shell only
