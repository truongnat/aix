# v0.5.0 Plan

## Purpose

Plan `v0.5.0` as a plugin-distribution-model release that clarifies how the harness should be consumed.

## Target Summary

`v0.5.0` should clarify the harness as a plugin-like markdown capability pack that can be installed, copied, or vendored into an existing repository without treating this source repository as the user's product repo.

## Scope

- plugin-like positioning
- distribution model docs
- consumption modes
- consumption mode guides
- release archive design
- installed surface clarity
- installed surface contract
- future runtime and plugin-registry boundaries

## Non-Goals

- no marketplace automation
- no runtime adapters
- no package publishing automation
- no server, database, or background systems
- no semantic validation
- no deep scanning

## Consumption Modes

- local install or copy into a target repo
- vendored harness directory
- global agent capability folder
- release archive
- future package or plugin registry

## Installed Surface

Clarify:

- what remains in the canonical source repository
- what should be installed into a target repository
- what agents are expected to read inside the target repository
- what should not be copied when the target repo only needs the operating surface

## Definition Of Done

- plugin-like positioning is explicit in the target docs and README
- source repo versus target repo distinctions are clear
- distribution model docs exist
- future plugin or registry modes are documented as optional channels
- `node bin/validate.js` passes
- `npm test` passes

## Release Risks

- introducing conceptual ambiguity instead of reducing it
- over-promising plugin distribution before the repo has a real distribution channel
- letting marketplace or runtime ideas creep into a docs-only clarification step

## Safety Boundaries

- keep markdown as the source of truth
- keep runtime systems out of scope
- do not add marketplace or package-publishing automation
- do not add semantic validation
- do not add deep repository scanning
- do not add secret scanning
- keep JavaScript changes limited to `install.js`, `validate.js`, or tests only when clearly justified
