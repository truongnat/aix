# v0.8.0 Readiness

## Purpose

Audit whether `v0.8.0` is ready to release as a Real Capability Pack Dogfood release.

## Target Summary

`v0.8.0` proves the plugin-like markdown capability pack works when consumed in real or semi-real target repositories with documented agent runtimes—not only when validated inside the source pack repository.

The release adds dogfood planning artifacts, two executed scenarios with reports, a friction log, follow-up backlog, dogfood-driven doc fixes, and an example-to-target layout guide on the path toward `v1.0.0`.

- real capability pack dogfood plan in `docs/v0.8.0-plan.md`
- dogfood scenarios in `docs/pack-dogfood-scenarios.md`
- dogfood report template in `docs/pack-dogfood-report-template.md`

- install.js `targetDisplay` for summary and next steps (no install behavior change)

## Dogfood Evidence

### Scenario A

- target: tiny local Node HTTP repo (external dogfood repo, not in source pack)
- runtime: Cursor (repo-local flow)
- goal: `health-check`
- install: `install.js` dry-run and write (83 paths, 0 failed)
- artifacts: full `.harness/` profile and `goals/health-check/` set
- validation: profile and goal structural validation **pass**

### Scenario B

- target: semi-real Flutter-style mobile skeleton (distinct from Scenario A)
- runtime: Codex (session discipline)
- goal: `google-login` (guest mode preserved in plan)
- install: same install flow as Scenario A
- artifacts: mobile-tuned profile and `goals/google-login/` set
- validation: profile and goal structural validation **pass**

### Minimum requirement

- **two distinct target repositories** exercised
- **two runtime patterns** documented: Cursor (A), Codex (B)
- **no v1 blockers** found in Scenario A or B

See [pack-dogfood-follow-up-backlog.md](pack-dogfood-follow-up-backlog.md).

## Remaining Gaps

- no Scenario C (backend/tooling third scenario) executed
- install surface still large (~83 paths) for tiny repos
- no minimal install tier implemented
- no archive generation, marketplace, or package publishing automation
- no runtime adapters
- no semantic validation or deep scanning
- example files under `examples/` not fully normalized to validator heading contracts (layout guide addresses adoption)

## Release Blockers

No release blockers identified.

Minimum dogfood evidence is present. Fix-pass items for Scenario A/B v0.8.x friction are addressed or explicitly partially addressed with documentation.

## Non-Blockers

These are intentional non-blockers for `v0.8.0`:

- no Scenario C
- large default install surface for tiny repos
- no minimal install tier
- no automation (archive generation, publishing, adapters)
- no semantic validation
- partial example-shape alignment without rewriting all example files

`v0.8.0` requires real dogfood evidence and clarity fixes, not distribution automation or contract freeze.

## Validation Status

Current validation status:

- `node validate.js` passes in the source pack
- Scenario A target: profile and goal validation passed (recorded in dogfood report)
- Scenario B target: profile and goal validation passed (recorded in dogfood report)
- fixture target profile and goal validation pass in CI tests

Validation remains structural-only.

## Test Status

Current test status:

- `npm test` passes (30 tests including install `targetDisplay` and PACK.md heading coverage)
- install dry-run, force, and summary behavior unchanged aside from display paths

## Safety Review

- markdown remains the source of truth
- no dependencies were added
- no `src/` directory was added
- no runtime adapters or distribution automation were added
- dogfood reports sanitize target repo details; external dogfood repos are not committed to the source pack
- no secrets or customer data in committed dogfood artifacts

## v0.8.0 Release Decision

`v0.8.0` is ready to release as a Real Capability Pack Dogfood release.

It demonstrates end-to-end pack consumption in two distinct target repositories with two runtime patterns and documents friction toward `v0.9.0` contract freeze and `v1.0.0` without expanding into automation.
