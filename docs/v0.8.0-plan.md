# v0.8.0 Plan

## Purpose

Plan `v0.8.0` as a Real Capability Pack Dogfood release that proves the harness works when consumed in real target repositories and agent runtimes—not only when validated inside the source pack repository.

## Target Summary

`v0.8.0` should produce evidence that install/copy consumption, runtime usage, harness profile creation, goal artifacts, and structural target validation work end-to-end outside `ai-engineering-harness`.

## Scope

- dogfood the pack in at least 2 target repositories (real or semi-real)
- test install or copy consumption from the source pack or a release snapshot
- test runtime consumption with at least 2 runtimes when possible
- create target `.harness/` profile artifacts in each dogfood repo
- create at least one goal artifact set per scenario
- run target profile and goal structural validation
- record friction in a structured log
- maintain a follow-up backlog classified by release impact

## Non-Goals

- no automation (install wizards, archive generation, CI dogfood runners)
- no runtime adapters
- no package or marketplace publishing
- no semantic validation
- no archive generation automation
- no changes driven only by source-repo polish without target-repo evidence

## Dogfood Targets

Minimum:

- 2 distinct target repositories (types may differ: tiny app, frontend/mobile, backend/tooling)
- at least 2 agent runtimes across scenarios when practical (for example Cursor + Claude Code, or Codex + Gemini CLI)

Preferred evidence per target repo:

- installed harness surface present
- `.harness/` profile artifacts created and reviewed
- one active goal under `.harness/goals/<goal-id>/`
- `node validate.js --target <path> --profile-only` passes
- `node validate.js --target <path> --goal <goal-id>` passes when a goal exists

## Dogfood Method

1. pick scenarios from [pack-dogfood-scenarios.md](pack-dogfood-scenarios.md)
2. install or copy the pack into the target repo per [consume-as-pack.md](consume-as-pack.md) and [installed-surface-contract.md](installed-surface-contract.md)
3. open the target repo in the chosen runtime; follow [runtime-consumption-model.md](runtime-consumption-model.md) and the runtime-specific guide
4. build or refine harness profile artifacts (`HARNESS.md`, `TEAM.md`, `SKILLS.md`, `WORKFLOW.md`, `GATES.md`, `MEMORY.md`)
5. create goal artifacts for the scenario goal id
6. run structural validation from the target repo (or with `--target` pointing at it)
7. fill [pack-dogfood-report-template.md](pack-dogfood-report-template.md) per scenario
8. log friction in [pack-dogfood-friction-log.md](pack-dogfood-friction-log.md)
9. triage follow-ups in [pack-dogfood-follow-up-backlog.md](pack-dogfood-follow-up-backlog.md)

## Evidence To Capture

- commands run (install, validate, agent prompts used)
- runtime and consumption mode
- artifacts created (paths and brief purpose)
- validation output (pass/fail and error text)
- what worked without explanation
- what was confusing or missing in docs
- pack surface gaps (install list, doc links, command discoverability)
- runtime-specific friction (workspace roots, wrong repo, path confusion)
- safety notes (secrets, wrong-repo work, over-copying source pack)

## Recommended Implementation Order

1. dogfood plan and scenario definitions (this step)
2. execute Scenario A (tiny repo / health-check)
3. execute Scenario B or C (second target repo type)
4. execute optional third scenario or second runtime on an existing repo
5. consolidate friction log and follow-up backlog
6. `v0.8.0` readiness and release (reports, not new automation)

## Definition Of Done

- at least 2 target repositories dogfooded with recorded reports
- at least 2 runtimes exercised when practical
- profile and goal validation evidenced per scenario
- friction log populated with classified items
- follow-up backlog triaged for `v0.8.x`, `v0.9`, `v1.0`, and later
- `node validate.js` and `npm test` still pass in the source pack
- no automation or adapter scope creep

## Release Risks

- dogfood repos are too similar to the source pack and hide real friction
- dogfood stops at install without profile/goal validation
- friction is logged but not classified, blocking `v0.9` contract freeze
- scope creep into adapters or automation to “fix” dogfood pain too early

## Safety Boundaries

- keep markdown as the source of truth
- do not store secrets, tokens, or private business data in dogfood artifacts committed to this repo
- summarize target-repo friction without preserving sensitive project details
- treat structural validation as necessary but not sufficient for “works in production”
- do not treat dogfood success inside this source repo as proof of external adoption
