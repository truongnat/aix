# v1.1.0 — Domain Skills, Spec Layer, And Prompt-Quality Pass

**Status:** minor release. New capabilities, all backward-compatible and opt-in. No breaking changes to existing command IDs, templates, or provider setup.

## Highlights

- **Agent-driven domain skill generation** — at `init`, an agent analyzes the project and returns a strict JSON profile; the harness gates that input and generates domain skills, path-scoped rules, and a domain workflow into the target repo. Detection is pure-agent (no brittle hardcoded heuristics); generation is deterministic and validator-safe.
- **Spec-driven layer (opt-in)** — delta specs (`CHANGE_SPEC.md` with `ADDED / MODIFIED / REMOVED Requirements`) and an optional durable `.harness/specs/` behavior spec, gated by config and off by default.
- **Prompt-quality standard** — every skill and dispatch prompt now follows a modular format (Role, Context, Task, Reasoning Procedure, Action Loop, Constraints, Examples, Output Format) embedding few-shot, chain-of-thought, and ReAct. Enforced by the validator and a conformance eval.
- **Delegated-worker memory (opt-in)** — workers can accumulate bounded notes across runs under `.harness/memory/workers/<agent>.md`, off by default.
- **Explorer worker + context-engineering doctrine** — a dedicated Explore-phase worker that returns a condensed map, plus `docs/context-engineering.md` documenting the five practices the harness implements.

## What changed

### Added

- **Domain skill generation** (`lib/domain-skill-generation.ts`, `lib/stack-detect.ts` analysis gate, `prompt-templates/domain-analysis.md`) — generates `.harness/skills/<domain>/`, auto-fills `.harness/SKILLS.md` and `.harness/WORKFLOW.md`, and writes path-scoped domain rules for **all four providers**: `.claude/rules/`, `.cursor/rules/`, `.codex/rules/`, and `.gemini/extensions/ai-engineering-harness/rules/`.
- **Spec layer** — `templates/CHANGE_SPEC.md`, `.harness/specs/`, and `specs` config block (`enabled: false` by default).
- **Worker memory** — `.harness/memory/workers/`, `workerMemory` config block (`enabled: false` by default), extended `record-subagent-result.js`.
- **Explorer worker** (`workers/explorer.md`) wired into the worker registry and `harness-map`.
- **Prompt format standard** (`skills/PROMPT_FORMAT_STANDARD.md`) + modular headings enforced across skills, capability prompts, and dispatch templates.
- **Context-engineering and token-budget docs** (`docs/context-engineering.md`, `docs/token-budget.md`).
- **Eval harness maturity** — A/B runner, LLM judge, scoring, extended metrics, and `aih insights` telemetry with CI eval-regression checks.
- **Policy engine** (`lib/policy/`) — programmable G0–G3 gates behind the guard hooks.

### Changed

- Backend lifecycle (install, uninstall, update, doctor, skeleton, git-hygiene) ported to in-process TypeScript for reliable provider-file writes.
- Generated domain skills conform to the full core skill heading contract, verified against the canonical heading list in tests.

## The loop is unchanged

```text
Session Start → Discuss → Plan → Run → Verify → Ship → Remember
```

Provider support posture is unchanged:

- Claude remains the strongest native path
- Cursor, Codex, and Gemini remain adapter or fallback paths (now including generated domain rules)

## Verification

- `npm test` — full contract + behavior suite (validator: 558 files/contracts)
- `node bin/validate.js`
- `npm run build` in `site/`
- `npm pack --dry-run`

## Upgrade notes

- No migration is required from `v1.0.x`.
- All new layers (`specs`, `workerMemory`, domain skills) are **opt-in** and default to off / empty — existing repos are unaffected until enabled.
- Existing command IDs, templates, and provider setup guidance remain the same.
- Release-facing docs and landing-page version surfaces stay synchronized with `package.json` via `scripts/sync-site-version.js`.
