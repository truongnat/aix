# v1.1.2 - Domain Bootstrap UX Fixes And Cursor Rules Cleanup

**Status:** patch release. No breaking changes to command IDs, provider support, or the core workflow loop.

## Highlights

- **Headless domain bootstrap polish** - `init` and `install` stay deterministic, while the agent-facing `harness-start` flow announces domain analysis visibly before generating skills.
- **Cursor install cleanup** - project installs no longer create a duplicate bare `rules/` directory at the repo root; only `.cursor/rules/` is written.
- **Release surface sync** - the package version, changelog, README badge, and docs index now point to `v1.1.2`.

## What changed

### Added

- `PLAN_DOMAIN_INIT_FIX.md` implementation follow-up now lands as a release commit, with the domain-bootstrap UX and install cleanup covered by tests.

### Changed

- `init` no longer prompts for domain analysis or domain selection.
- `install` no longer asks for mode / init / cache / proceed prompts during the primary wizard flow.
- `commands/harness-start.md` now includes an explicit visible domain-bootstrap step for first-session setup.

### Fixed

- Cursor project installs no longer emit a duplicate bare `rules/` folder alongside `.cursor/rules/`.
- The repository now has a fresh patch release and matching release notes surface for the next npm publish.

## The loop is unchanged

```text
Session Start → Discuss → Plan → Run → Verify → Ship → Remember
```

## Verification

- `npm run build`
- `npm test`
- `node bin/validate.js`

## Upgrade notes

- No migration is required from `v1.1.1`.
- Existing `domains` selection, analysis parsing, and provider install behavior remain backward-compatible.
- The new headless flow stays additive; interactive project setup still works for provider selection.
