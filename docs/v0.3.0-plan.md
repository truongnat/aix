# v0.3.0 Plan

## Purpose

`v0.3.0` is planned as Lightweight Host Repository Validation.

The goal is to extend the existing dependency-free validator so it can validate an adopted host repository structurally, without introducing runtime complexity or drifting into implementation judgment.

## v0.3.0 Target

Add a minimal target-repository mode:

- `node validate.js --target <path>`
- optional `--profile-only`
- optional `--goal <goal-id>`

The validator should check only structural `.harness/` contracts and reuse existing heading contracts where practical.

## Target Repo Validation Scope

`v0.3.0` should validate:

- host repository root prerequisites
  - `AGENTS.md`
  - `.harness/`
- required harness profile artifacts
  - `.harness/HARNESS.md`
  - `.harness/TEAM.md`
  - `.harness/SKILLS.md`
  - `.harness/WORKFLOW.md`
  - `.harness/GATES.md`
  - `.harness/MEMORY.md`
- optional project context artifacts as warnings or informational checks
  - `.harness/PROJECT.md`
  - `.harness/STATE.md`
  - `.harness/CONTEXT.md`
  - `.harness/ROADMAP.md`
- optional goal-level artifacts when `--goal <goal-id>` is requested
  - `.harness/goals/<goal-id>/GOAL.md`
  - `.harness/goals/<goal-id>/PLAN.md`
  - `.harness/goals/<goal-id>/TASKS.md`
  - `.harness/goals/<goal-id>/VERIFY.md`
  - `.harness/goals/<goal-id>/REMEMBER.md`

## CLI Behavior

Current Step 3 commands:

- `node validate.js`
  - validate this harness repository itself, using the current behavior
- `node validate.js --target ../my-project`
  - validate an adopted host repository's root and harness profile artifacts
- `node validate.js --target ../my-project --profile-only`
  - validate only root and profile artifacts, skip goal checks
- `node validate.js --target ../my-project --goal google-login`
  - validate root and profile artifacts, then validate the named goal artifact set

Current Step 3 behavior rules:

- `--target` changes the validation base directory
- `--profile-only` and `--goal` only make sense with `--target`
- `--target <path>` defaults to profile validation
- if both `--profile-only` and `--goal` are provided, the command should fail with a clear usage error
- no recursive deep scanning beyond the expected harness paths


## Safety Boundaries

`v0.3.0` must remain safe and lightweight:

- no dependencies
- no markdown parser
- no runtime adapters
- no server, database, Docker, LangGraph, or background workers
- no deep repository scanner
- no secret detection engine
- no semantic correctness judging
- no arbitrary large-file scanning
- never print secrets or sensitive file contents

Validation should remain structural and contract-focused.

## Reuse Of Existing Contracts

Where practical, `v0.3.0` should reuse existing heading contracts already defined in `validate.js`:

- harness profile headings
- goal artifact headings
- existing required path patterns

The implementation should avoid introducing a second inconsistent contract system.

## Implementation Steps

Step 1 note:

- refactor the validator into reusable helpers and CLI parsing first
- keep default repository-self validation behavior unchanged
- return a clear usage error for `--target` until target validation is implemented

Step 2 note:

- implemented target profile validation first
- `--target <path>` now runs profile validation by default
- `--target <path> --profile-only` is the explicit equivalent

Step 3 note:

- implemented target goal validation second
- `--target <path> --goal <goal-id>` now validates profile artifacts plus the requested goal artifact set
- validation remains structural-only and scoped to the requested goal path

Step 4 note:

- documentation polish is the current step
- target validation usage docs now explain the implemented commands and their structural-only boundary
- profile validation and goal validation are both implemented and documented for safe use

Implementation sequence:

1. add lightweight argument parsing for `--target`, `--profile-only`, and `--goal`
2. preserve current repository-self validation as the default mode
3. extract shared helpers for path existence, non-empty checks, and heading checks
4. add a small target-repo validation path set for root and profile artifacts first, then extend later for optional context and optional goal artifacts
5. reuse existing heading arrays for profile and goal artifacts where possible
6. add clear usage errors for invalid argument combinations
7. add fixture-based tests for:
   - valid target repo profile
   - missing profile files
   - valid goal artifact set
   - missing or invalid goal artifacts
   - invalid CLI flag combinations
8. keep output flat, short, and deterministic

## Non-Goals

`v0.3.0` should not add:

- runtime adapters
- server code
- database-backed memory
- Docker
- LangGraph
- deep repository scanning
- semantic validation of application behavior
- package publishing automation

## Current Status

Step 3 now covers target goal validation on top of target profile validation. Validation remains structural and does not scan arbitrary repository files.
