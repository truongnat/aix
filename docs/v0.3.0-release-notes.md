# v0.3.0 Release Notes

## Summary

`v0.3.0` releases Lightweight Host Repository Validation.

This version extends the existing dependency-free validator so it can validate adopted host repository harness artifacts without drifting into runtime systems or semantic application judgment.

## What Is Included

- lightweight host repository validation
- `node validate.js --target <path>`
- `node validate.js --target <path> --profile-only`
- `node validate.js --target <path> --goal <goal-id>`
- target profile validation
- target goal validation
- valid and invalid fixtures plus tests for target validation flows
- target validation usage documentation
- `v0.3.0` readiness and release scope documentation

## What Changed From v0.2.0

`v0.2.0` documented the target validation direction.

`v0.3.0` implements that direction as a lightweight structural validator for:

- this harness repository
- adopted host repository profile artifacts
- adopted host repository goal artifact sets

## What Is Intentionally Not Included

- semantic application correctness checks
- application code validation
- secret detection engine
- runtime adapters
- server, database, or background systems
- markdown parser
- deep repository scanner
- package publishing automation

## Who Should Use This Release

Use `v0.3.0` if you:

- adopt `ai-engineering-harness` into another repository
- want a dependency-free check for required `.harness/` profile artifacts
- want a structural check for `.harness/goals/<goal-id>/` artifacts
- want validation that stays markdown-first and runtime-light

## How To Validate This Harness Repository

Run:

```bash
node validate.js
```

This checks the harness repository's required documentation, templates, examples, and validation contracts.

## How To Validate An Adopted Host Repository Profile

Run either:

```bash
node validate.js --target ../my-project
node validate.js --target ../my-project --profile-only
```

These commands validate:

- `AGENTS.md`
- `.harness/`
- `.harness/HARNESS.md`
- `.harness/TEAM.md`
- `.harness/SKILLS.md`
- `.harness/WORKFLOW.md`
- `.harness/GATES.md`
- `.harness/MEMORY.md`

## How To Validate An Adopted Host Repository Goal

Run:

```bash
node validate.js --target ../my-project --goal google-login
```

This validates the target profile first, then validates:

- `.harness/goals/google-login/GOAL.md`
- `.harness/goals/google-login/PLAN.md`
- `.harness/goals/google-login/TASKS.md`
- `.harness/goals/google-login/VERIFY.md`
- `.harness/goals/google-login/REMEMBER.md`

## Known Limitations

- validation is structural-only
- validation does not prove application correctness
- validation does not inspect arbitrary application files
- validation does not scan for secrets across the repository
- validation does not replace human review

## Next Likely Milestone

The next likely milestone is incremental polish around documentation and optional validation ergonomics, while keeping the harness markdown-first and runtime-light.
