# v0.4.0 Plan

## Purpose

Plan `v0.4.0` as an ergonomics-focused release that improves first-time adoption experience after the `v0.3.x` dogfood findings.

## Target Summary

`v0.4.0` should make install-to-profile flow clearer without adding runtime complexity.

The release should stay markdown-first, dependency-free, and structural in spirit while reducing first-use friction.

## Scope

- clearer post-install guidance
- compact installer output examples
- improved install-to-profile docs
- optional validation ergonomics improvements
- richer tiny-repo adoption example
- memory conventions for very small repositories

## Non-Goals

- runtime adapters
- semantic validation
- server, database, or background systems
- large skill catalog import
- deep scanner
- secret detection engine
- package publishing automation

## Candidate Improvements

- add a clearer “what to do next” path immediately after install
- add compact examples that show expected install output shape
- tighten adoption docs around first profile creation
- make tiny-repo examples easier to copy and adapt
- document lighter memory conventions for small or toy repositories
- consider small validation ergonomics improvements that do not expand validation scope

## Recommended Implementation Order

1. improve post-install guidance in adoption-facing docs
2. add compact installer output examples
3. improve install-to-profile walkthrough clarity
4. add a richer tiny-repo adoption example
5. document lightweight memory conventions for small repositories
6. evaluate small validation ergonomics improvements only if they preserve the structural-only boundary

Current step:

- Step 1 covers post-install guidance and installer output examples
- Step 2 adds a tiny-repo adoption example
- Step 3 adds small-repo memory conventions
- Step 4 adds a compact install-to-profile walkthrough
- Step 5 adds a validation troubleshooting guide

## Definition of Done

- post-install next steps are clearer for first-time users
- install-to-profile flow is easier to follow in docs
- tiny-repo adoption is better illustrated
- small-repo memory guidance is explicit
- validation and install behavior remain runtime-light
- `node validate.js` passes
- `npm test` passes
- no dependencies are added
- no `src/` directory is added

## Release Risks

- over-correcting into feature expansion instead of ergonomics polish
- adding too much installer complexity for a minor usability gain
- blurring the boundary between structural validation and semantic validation

## Safety Boundaries

- keep markdown as the source of truth
- do not add runtime platforms or adapters
- do not add semantic application validation
- do not add deep repository scanning
- do not add secret detection systems
- keep JavaScript changes limited to `install.js`, `validate.js`, or tests only when clearly justified
