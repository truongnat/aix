# v0.7.0 Plan

## Purpose

Plan `v0.7.0` as a Capability Pack Packaging release that defines a lightweight pack manifest and packaging contract.

The release makes the harness easier to identify, verify, and distribute manually as a versioned capability pack without adding marketplace or publishing automation.

## Target Summary

`v0.7.0` should define what a pack artifact is, how pack metadata is expressed, how the installed surface maps to a distributable pack, and how users verify a pack manually before adoption.

## Scope

- define pack manifest format and spec
- add `PACK.md` as the canonical manifest
- define pack metadata and included-surface mapping
- document manual release archive packaging
- add pack verification checklist
- optionally add lightweight manifest heading validation if it remains simple and dependency-free

## Non-Goals

- no npm publishing
- no marketplace automation
- no runtime adapters
- no compiled plugin package
- no archive generation automation unless explicitly deferred
- no server, database, or background systems
- no semantic validation
- no deep repository scanning

## Packaging Concepts

- **source pack**: this repository as the canonical authored pack
- **installed surface**: the subset copied into a target repository for agent operation
- **release archive**: a versioned snapshot of the pack surface for manual distribution
- **pack manifest**: `PACK.md` describing pack identity, version, surface, and validation expectations
- **pack version**: the released or planned harness version the manifest describes
- **supported consumption modes**: install/copy, vendored directory, global capability folder, release archive, future registry
- **runtime compatibility notes**: docs-only guidance for Claude Code, Cursor, Codex, Gemini CLI, OpenCode

## Recommended Implementation Order

1. pack manifest design (`docs/pack-manifest-spec.md`, `docs/packaging-model.md`)
2. pack manifest file (`PACK.md`)
3. pack verification checklist
4. optional lightweight manifest validation
5. manual packaging guide
6. readiness and release

Current step:

- Step 2 added [pack-verification-checklist.md](pack-verification-checklist.md)
- Step 3 added lightweight `PACK.md` heading validation in `validate.js`
- Step 4 added [manual-packaging-guide.md](manual-packaging-guide.md)

## Definition Of Done

- packaging model and manifest spec exist
- `PACK.md` exists with required sections
- distribution docs reference `PACK.md` for archives and consumption modes
- pack verification checklist exists
- manual packaging guidance is documented
- `node validate.js` passes
- `npm test` passes

## Release Risks

- manifest drift from actual installed surface or `install.js` behavior
- over-specifying packaging and drifting toward npm or marketplace automation
- treating `PACK.md` as required in every target repo when it is mainly a source-pack and archive artifact

## Safety Boundaries

- keep markdown as the source of truth
- keep packaging docs-only unless simple manifest validation is added
- do not add runtime adapters, marketplace automation, or package publishing automation
- do not add semantic validation or deep scanning
- keep structural validation separate from pack identity checks
