# v0.5.0 Release Notes

## Summary

`v0.5.0` is a Plugin Distribution Model release.

It clarifies `ai-engineering-harness` as a plugin-like markdown capability pack instead of a cloned working project.

## What Is Included

- plugin-like capability pack positioning
- `TARGET.md` pivot
- plugin model documentation
- distribution model documentation
- installed surface contract
- consumption mode guide
- consume-as-pack quick guide
- release archive model
- release archive checklist
- `v0.5.0` readiness and release scope docs
- clearer source repo vs target repo distinction

## What Changed From v0.4.0

`v0.4.0` focused on adoption ergonomics inside the existing install-to-profile flow.

`v0.5.0` changes the framing around that flow:

- the source repository is now explicitly the canonical pack source
- the target repository is explicitly the real product work tree
- the installed surface is documented as a deliberate subset
- consumption modes are explained as capability-pack modes rather than clone-and-work-here usage
- release archives are defined as a future-friendly distribution model without implementing automation

## What Is Intentionally Not Included

- marketplace automation
- package publishing automation
- release archive generation
- runtime adapters
- server, database, or background systems
- semantic validation
- secret scanning
- deep repository scanning

## Who Should Use This Release

Use `v0.5.0` if you want:

- a clear capability-pack mental model
- explicit source-repo versus target-repo boundaries
- a documented installed surface contract
- practical consumption-mode guidance
- a future-friendly release-archive model without runtime complexity

## How To Consume The Harness As A Pack

Treat this repository as the canonical source pack.

Use the current supported flow:

```bash
node install.js --target ../my-project --dry-run
node install.js --target ../my-project
node validate.js --target ../my-project --profile-only
node validate.js --target ../my-project --goal health-check
```

Then use the harness from inside the target repository, not inside the source pack.

## Known Limitations

- no marketplace or package publishing automation exists
- no release archive generation exists
- no runtime adapters exist
- no global agent capability folder integration exists
- validation remains structural-only

## Next Likely Milestone

The next likely milestone is a follow-up that decides whether the distribution model should stay docs-only a bit longer or move into carefully scoped archive-generation or packaging support without losing the markdown-first boundary.
