# v1.1.3 - Provider Binary Detection And Manifest-Aware Lifecycle

**Status:** patch release. No breaking changes to command IDs, provider support, or the core workflow loop.

## Highlights

- **Provider binary detection** - `install` and `init` now gate interactive setup on real provider CLIs instead of repo residue, so missing binaries fail fast.
- **Manifest-aware lifecycle** - `update`, `uninstall`, `status`, and `doctor` now prefer the recorded install manifest instead of guessing from target-directory heuristics.
- **Release surface sync** - the package version, changelog, README badge, and docs index now point to `v1.1.3`.

## What changed

### Added

- Provider CLI probing for `claude`, `codex`, `cursor`, and `gemini`, including `cursor-agent` fallback detection for Cursor.
- A new release changeset so the version bump follows the repo’s own changesets workflow.

### Changed

- `init` now defaults to detected provider CLIs and fails fast when none are installed.
- `install` now marks provider selection by binary availability and hard-stops project installs outside git.
- `update` and `uninstall` now respect the recorded `.ai-harness/manifest.json` surface.
- `status` now reports provider binaries and manifest providers; `doctor` now checks manifest readability.

### Fixed

- Project installs now stop before writing when the target directory is not a git repo.
- Provider updates no longer drift off the manifest-recorded install surface.

## The loop is unchanged

```text
Session Start → Discuss → Plan → Run → Verify → Ship → Remember
```

## Verification

- `npm run build`
- `npm test`
- `node bin/validate.js`

## Upgrade notes

- No migration is required from `v1.1.2`.
- Existing install surfaces remain backward-compatible.
- The new binary gate only changes how provider availability is detected before installation.
