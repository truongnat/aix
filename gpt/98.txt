# 🚀 PHASE 19

# Sandboxed Tool Execution + Capability Security Model 🔐

Now we harden it.

Up to now:

Skills are trusted code.

Phase 19 introduces:

* Capability isolation
* Resource sandboxing
* Execution permission model
* Runtime safety guarantees

This is where we prevent:

* Arbitrary file writes
* Network exfiltration
* Privilege escalation
* Tool misuse via planner synthesis

---

# 🎯 Objective

Make each skill explicitly declare:

* What it is allowed to do
* What resources it can access
* What side effects it can produce

And enforce it.

---

# 🔥 STEP 91 — Capability Permission Model

Extend SkillCapability:

```rust
pub struct CapabilityPermissions {
    pub allow_fs_read: bool,
    pub allow_fs_write: bool,
    pub allow_network: bool,
    pub allow_env: bool,
    pub allow_process_spawn: bool,
}
```

Then:

```rust
pub struct SkillCapability {
    ...
    pub permissions: CapabilityPermissions,
}
```

No defaults. Explicit only.

---

# 🔥 STEP 92 — Runtime Permission Guard

Executor must:

* Wrap skill execution in a permission guard
* Enforce declared permissions
* Abort if violation detected
* Emit `SecurityViolation` event

If skill attempts disallowed action → hard fail.

---

# 🔥 STEP 93 — Domain-Level Policy Override

Add:

```rust
pub struct DomainSecurityPolicy {
    pub override_permissions: Option<CapabilityPermissions>,
}
```

Allows:

* Restrict domain globally
* Harden specific environments
* Disable network globally

CLI:

```bash
--disable-network
--read-only
```

---

# 🔥 STEP 94 — Deterministic Permission Validation

Validator must:

* Ensure skill permissions comply with routing policy
* Reject plan if selected skill violates global policy

This prevents planner from choosing unsafe skill.

---

# 🔥 STEP 95 — Side-Effect Classification

Add to SkillCapability:

```rust
pub enum SideEffectClass {
    Pure,
    Idempotent,
    ExternalMutation,
}
```

Planner may:

* Prefer Pure under strict mode
* Penalize ExternalMutation in cost model

Deterministically.

---

# 🔥 STEP 96 — Snapshot Security Integrity

Snapshot must persist:

* Security policy
* Permission state
* No privilege escalation during resume

Resume must NOT allow broader permissions than original.

---

# 🔥 STEP 97 — Execution Sandboxing

If feasible:

* Run external skills in subprocess
* Apply OS-level sandbox (ulimit / seccomp / chroot simulation)
* Capture stdout/stderr
* Enforce timeout

Deterministic timeout failure → `SecurityViolation`.

---

# 🔥 STEP 98 — Security Test Matrix

Test:

1. Skill tries FS write without permission → fail
2. Skill allowed FS read only → pass
3. Domain-level network disabled overrides skill permission
4. Snapshot resume preserves security policy
5. Planner avoids unsafe skill when safe alternative exists

All deterministic.
