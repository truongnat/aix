`.agent/state/repo.lock`

CLI → Planner → ExecutionEngineV2

### 3️⃣ Panic guard wrapper (`tokio::spawn` + join)

Cái này cực kỳ quan trọng.

Bạn đã chuyển:

```
panic → controlled Err
```

Thay vì:

```
panic → crash process → orphan lock file
```

Đây là runtime hardening thực sự.

# 🚨 Những điểm còn lại trước khi gọi COMPLETED

Giờ chỉ còn 4 thứ thực sự quan trọng.

---

## 1️⃣ Stale Lock Resolution (Chưa rõ đã có chưa)

Repo lock hiện tại:

```
.agent/state/repo.lock
```

Câu hỏi quan trọng:

* Lock file có lưu PID không?
* Có check PID còn sống không?
* Nếu process crash trước khi release lock thì sao?

Nếu chưa có stale detection:

Bạn có nguy cơ:

> Engine bị deadlocked vĩnh viễn sau crash.

### Production-ready cần:

```json
{
  "pid": 12345,
  "started_at": "...",
  "hostname": "..."
}
```

Và logic:

* Nếu PID không tồn tại → reclaim lock
* Hoặc nếu lock quá X phút → override với warning

---

## 2️⃣ Idempotency Layer (Vẫn là lỗ lớn nhất)

Hiện tại:

Resume dựa vào:

* step state
* execution result

Nhưng không verify external world.

Ví dụ:

* Step: git commit
* Crash sau commit, trước state save

Resume:

* commit lại lần nữa

Nếu bạn chưa có detect_already_applied → vẫn chưa deterministic thật sự.

Đây là thứ duy nhất mình còn coi là “chưa hoàn chỉnh”.

---

## 3️⃣ Determinism Integration Tests

Bạn có unit tests (36 pass).

Nhưng cần:

### Test A: Crash mid-step

* Inject artificial panic sau side-effect
* Restart engine
* Resume
* Assert no double side-effect

### Test B: Order determinism

Nếu có DAG:

* Multiple ready steps
* Order phải stable (lexicographic id hoặc defined sort)

Nếu chưa có integration test loại này → determinism chưa được chứng minh.

---

## 4️⃣ Sandbox Isolation Depth

Bạn có timeout.

Nhưng mình chưa thấy xác nhận:

* env_clear()
* command whitelist
* working directory isolation
* resource limits

Nếu run_script vẫn inherit full environment → engine chưa thực sự safe.
