# ⚠️ 2️⃣ Những lỗ hổng tiềm ẩn

Dù mọi thứ pass test, có 4 vùng cần chú ý:

---

## ⚠️ A. Workflow Execution Safety

`agent.run_script` hiện tại:

* Có sandbox không?
* Có timeout?
* Có whitelist command?
* Có environment isolation?

Nếu không:

Bạn đang cho workflow quyền full shell execution.

Ở local dev thì ok.
Ở CI / SaaS agent thì cực nguy hiểm.

👉 Phase tiếp theo nên thêm:

* Timeout guard
* Allowed command list
* Capture stdout/stderr structured
* Exit code mapping

---

## ⚠️ B. Workflow Determinism

YAML workflow hiện tại:

* Có conditional steps không?
* Có retry?
* Có failure handling strategy?

Ví dụ:

```yaml
- run: cargo test
  on_failure: abort
```

Hiện tại nếu script fail:

* Workflow có dừng không?
* Có rollback không?

Opinionated workflow mà không có failure strategy thì chưa hoàn chỉnh.

---

## ⚠️ C. State Model

Hiện tại workflow có:

* session thread
* branch orchestration

Nhưng có:

* workflow state persistence?
* step resume?
* idempotency?

Nếu workflow crash giữa chừng:
→ Có resume được không?

Nếu không thì Phase 23 vẫn là synchronous runner.

---

## ⚠️ D. Rules chưa được enforce thật sự

Bạn có:

```
.agent/rules/*.yaml
```

Nhưng:

* Rules có được validate tự động không?
* Có linter check branch naming không?
* Có coding_rules enforcement không?

Nếu rules chỉ được load mà không enforce runtime thì mới là “config”, chưa phải “policy engine”.

# 🚀 3️⃣ Phase 24 nên làm gì?

Nếu mình là bạn, mình sẽ đặt tên:

# PHASE 24 — Deterministic Workflow Engine

Mục tiêu: biến hệ thống từ “workflow runner” thành “orchestrator”.

---

## 🔥 Hướng 1: Add Failure Semantics

Mở rộng YAML spec:

```yaml
steps:
  - run: cargo fmt
    on_failure: continue

  - run: cargo test
    on_failure: abort

  - run: deploy.sh
    retry: 3
```

Engine cần:

* StepResult enum
* Retry logic
* Fail-fast mode
* Failure hooks

---

## 🔥 Hướng 2: Add Execution Sandbox Layer

`run_script` nên có:

* timeout
* resource limit
* command policy
* working directory isolation

Nếu muốn lên SaaS:
→ bắt buộc phải làm.

---

## 🔥 Hướng 3: Workflow State Machine

Hiện tại bạn có:

```
Workflow → Steps → Skills
```

Nên nâng cấp thành:

```
WorkflowInstance
  ├── current_step
  ├── status
  ├── step_results[]
  └── persisted_state.json
```

Khi crash:
→ resume từ step N.

Đây là bước nhảy từ tool sang platform.

---

## 🔥 Hướng 4: Policy Enforcement Engine

Hiện bạn có rules YAML.

Nâng cấp thành:

```
RuleEngine
  - validate_branch_name()
  - validate_commit_message()
  - validate_pr_title()
  - validate_code_patterns()
```

Opinionated system mà không enforce rule → chỉ là guideline.

---

## 🔥 Hướng 5: CLI UX nâng cấp

Ví dụ:

```
agent workflow run feature --dry-run
agent workflow resume
agent workflow status
agent workflow graph
```

Graph visualization đặc biệt quan trọng nếu sau này bạn thêm conditional branching.
