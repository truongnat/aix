# Brainstorming — Form-Filling Guide

How to fill `assets/brief-template.md` so it satisfies the `Output Contract` in `SKILL.md`.

## Problem Statement

One or two sentences, in concrete engineering terms. Not the user's original wording — the restated, scoped version.

## Constraints

Bullet list. Only confirmed constraints (repo conventions, must-not-break contracts, provider compatibility). Do not list assumptions here.

## Success Criteria

Bullet list. How the eventual implementation will be judged done. Must be checkable, not aspirational.

## Options Considered

Two or three realistic approaches max, one line each: name + one-sentence tradeoff. Do not pad with weak options just to look thorough.

## Recommendation

One option, named explicitly, with a one-sentence reason it fits this repo specifically (not a generic best practice).

## Handoff

State explicitly what reads this brief next — a planning step, a human reviewer, or another agent turn. Don't leave it implied.

## Validation

A filled brief is acceptable only if every section above is non-empty and the Recommendation names exactly one option.
